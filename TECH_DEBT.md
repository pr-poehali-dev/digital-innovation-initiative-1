### 0.1. 🟡 Долгоживущая тест-сессия в БД (временное решение)
- **2026-05-24:** для regression-тестов изоляции используется session_id `regression_test_session_userb_isolation_check_64hexchar000000aaa` с `expires_at = NOW() + 10 years`, привязанный к user_id=2.
- **Зачем:** tests.json не поддерживает динамический login перед каждым тестом — нужна стабильная сессия для проверки «User B → 403 на чужие данные».
- **Риски:** если session_id утечёт — постоянный доступ от имени тестового юзера; тесты завязаны на конкретное состояние БД; нельзя гонять параллельно несколько окружений.
- **Целевое решение (TODO в следующих спринтах):**
  1. Отдельный test environment (отдельная схема БД для тестов)
  2. Bootstrap-логин перед тестами через test-only endpoint с whitelisted IP
  3. Или CI-окружение с краткоживущей сессией (5 мин), создаваемой через сервисный аккаунт
- **Митигация сейчас:** test user B не имеет прав ни в одном реальном проекте, его сессия даёт 403 на все продакшен-данные.

### 2. RBAC примитивный
- ✅ Есть роли `owner` / `member` в БД
- ✅ `project.invite` и `project.update` — только owner
- ❌ Нет ролей `editor` / `reviewer` / `viewer`
- ❌ Документы не имеют прав доступа сверх проекта
- ❌ Задания не имеют прав доступа сверх проекта
- ❌ Нет матрицы «кто что может» в коде

### 3. Изоляция данных не покрыта тестами
- ✅ В коде есть `check_access` в `projects.handle_get/update/invite`
- ❌ В `tasks`, `documents`, `generate` проверка дублируется в каждом handler-е, легко забыть
- ❌ Нет автоматического теста «пользователь B не видит проект A»
- ❌ Нет проверки direct-доступа к файлам S3 в обход бэкенда

### 4. Безопасность не документирована
- ✅ Пароли хешируются SHA-256
- ❌ Нет соли в хеше паролей (миграция нужна)
- ❌ Нет rate limiting на login
- ❌ Нет защиты от CSRF (cookies не используются — пока не критично, но при переходе на cookies станет проблемой)
- ❌ Нет аудита access_denied событий
- ❌ Файлы в S3 — публичные по CDN URL (нет проверки прав при скачивании)

### 5. Поиск — НЕ настоящий RAG
- ✅ Чанкование документов работает
- ✅ Keyword search по чанкам работает
- ❌ Нет embeddings
- ❌ Нет pgvector / pinecone
- ❌ Нет reranking
- ❌ Релевантность — простой счёт вхождений ключевых слов

### 6. Веб-поиск через HTML-скрейпинг DuckDuckGo
- ✅ Работает как fallback
- ❌ Может сломаться в любой момент при изменении HTML
- ❌ Нет контроля качества источников
- ❌ Может нарушать ToS DuckDuckGo
- **Решение:** перейти на Yandex Search API когда будет ключ
