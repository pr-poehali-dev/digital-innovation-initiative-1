# ProjectPage: Stage 7–11 Impact

## One-pager

**Что это:** Управленческий обзор полигона трансформации перестал быть просто дашбордом с цифрами и превратился в рабочий инструмент, который сам доводит пользователя от «увидел проблему» до «проблема решена» — без ручного поиска, ручной фильтрации и повторного захода в интерфейс.

**Зачем это бизнесу:** Руководитель тратит меньше времени на «а что там опять зависло» и больше — на само решение. Каждое узкое место в трансформации (инициатива без владельца, гипотеза без инициативы) теперь закрывается за один непрерывный поток действий.

**Масштаб изменений:** 5 этапов (Stage 7–11), только фронтенд, без изменений в базе данных и бэкенде — вся логика переиспользует существующие API.

## Before / After

**До Stage 7–11:**
- Обзор показывал проблему, но дальше пользователь сам искал нужную вкладку
- Сам фильтровал список глазами — фильтров не было вообще
- Сам пытался понять причину, вчитываясь в поля карточки
- Сам открывал объект и редактировал его через общую форму
- После сохранения сам заново искал, что делать дальше — интерфейс не подсказывал

**После Stage 7–11:**
- Клик по виджету обзора сразу открывает нужную вкладку с применённым фильтром
- Список сам объясняет причину прямо на карточке (бейджи: «Нет владельца», «Нет следующего шага», «Готово к запуску», «Без инициативы»)
- Карточка даёт точную кнопку действия под конкретную причину
- После сохранения список обновляется сам, без перезахода
- Очередь доводится до понятного завершения — «Все зависшие инициативы обработаны»

## Changelog Stage 7–11

**Stage 7 — Preset-навигация из виджетов.** Клик по виджету обзора открывает нужную вкладку с примененным фильтром через URL (`?tab=initiatives&preset=stalled`), с чипом и возможностью сброса.

**Stage 8 — Объясняющий drilldown.** На карточках появились бейджи-причины («Нет владельца», «Готово к запуску», «Без инициативы») и осмысленные пустые состояния вместо универсального «ничего не найдено».

**Stage 9 — Контекстное действие.** На карточке — точная кнопка действия под конкретную причину («Указать владельца», «Заполнить владельца и следующий шаг»), открывающая компактную форму на 1–2 поля без inline-редактирования всей карточки.

**Stage 10 — Режим очереди.** Детерминированная сортировка карточек (сначала самые проблемные), индикатор «Осталось: N», completion-состояние при полной обработке.

**Stage 11 — Плавная передача внимания.** После действия — короткий success-feedback, автоматический переход и подсветка следующей карточки в очереди, защита от повторного клика во время сохранения.

## Demo Runbook

**Проект:** «Создание Управления» (workspace_mode = polygon)

**Основной live-demo путь:** через preset `stalled` — единственный сценарий, гарантированно дающий 1 реальную карточку с обеими причинами и completion-state сразу после исправления.

**Что должно быть заполнено заранее:** ничего готовить не нужно — инициатива «Ручной ввод 2» (id=2, статус `approval`, приоритет `high`, `owner_name` и `next_step` пустые) уже в нужном состоянии.

**Формулировка для инвестора (важно):** говорить «у неё не указан ни владелец, ни следующий шаг», а не просто «без ответственного» — это синхронно с обоими бейджами на карточке.

### Если объект уже исправлен кем-то до демо

1. Проверить `preset=stalled` — если 0 карточек, переключиться на fallback
2. **Fallback 1:** виджет «Готовы к запуску» (`launch_ready`) — показать бейдж «Готово к запуску» на любой инициативе со статусом `preparation/approval/in_plan` с заполненными полями
3. **Fallback 2:** виджет «Гипотезы без инициатив» (`without_initiative`) — создать гипотезу из любой из 4 проблем кейса через кнопку «Создать гипотезу», не доводя до инициативы
4. **Fallback 3 (гарантированный):** создать новую инициативу через «Добавить» с пустыми owner/next_step и статусом `approval` — за 20 секунд получаем demo-объект

### Post-demo reset (обязательно)

Live-demo изменяет данные: после показа карточка исчезает из `stalled`. Перед следующим прогоном демо нужно:

- либо вернуть `owner_name=''` и `next_step=''` у инициативы «Ручной ввод 2»,
- либо создать новый demo-объект (см. Fallback 3) перед следующим показом.

Без этого шага следующий прогон демо может «сломаться» просто потому, что предыдущий показ был успешным.

### Важно для product-demo

Эффект «фокус переходит на следующую карточку» в текущих данных не проявляется (в очереди только 1 карточка) — упоминать его как продуктовое поведение при нескольких карточках в очереди, а не как то, что будет видно live на этом seed-датасете.
