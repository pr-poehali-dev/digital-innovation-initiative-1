# ADR-0001: База практик, автоматизаций и решений

Статус: **принято как каркас (design note), реализация отложена**
Дата: 2026-07-09
Контекст: «Траектория» — цифровизация функций подразделений.

## Проблема
Нужен отдельный контур-справочник: какие практики и классы автоматизации существуют,
какое ПО и вендоры (РФ/иностранные) их закрывают, чтобы **подбирать решения к функциям и
процессам**, а не хранить «таблицу вендоров».

## Решение (архитектура, вглубь — не вширь)
Строить как связанный контур, НЕ связывать функцию напрямую с вендором:

```
function/process → practice → capability → product/module → vendor/company → evidence
```

Причины:
- функцию можно улучшить и без ПО (оргпрактика);
- один capability закрывают десятки продуктов;
- вендор покрывает лишь часть capability;
- разные модули одного вендора сильно отличаются.

## Сущности (минимальные)
- **Practice** — практика улучшения (в т.ч. не цифровая): единая очередь, база знаний, чек-листы, второй контроль, SLA-мониторинг, типизация кейсов, эскалации.
- **Capability** — что должна уметь автоматизация: маршрутизация, OCR/IDP, rules engine, case management, knowledge search, аудит, отчётность, screening/scoring, генерация черновиков, контроль сроков.
- **ProductModule** — конкретный продукт/модуль/линейка + deployment model + подтверждённые capability.
- **VendorCompany** — компания (разработчик / интегратор / консалтинг / MSP) — разделять роли.
- **Offer** — что предлагает компания: лицензия / внедрение / кастомизация / сопровождение / аутсорсинг.
- **Evidence** — источник любого утверждения (сайт, документация, datasheet, кейс, реестр, интервью, пилот).
- **Mapping** — связь practice/capability/product ↔ функция/процесс: применимость, ожидаемый эффект, ограничения, основание.

## Правила качества (source-backed)
1. Любое утверждение о продукте — только с источником (Evidence).
2. Разделять типы источника: заявление вендора / независимый источник / внутренний опыт / пилот.
3. Хранить дату актуальности.
4. Не хранить «лидер/лучший» без методики.
5. Capability помечать: **из коробки / с доработкой / через партнёра**.
6. Не смешивать ПО и услуги внедрения (ProductModule ≠ Offer).

## Слои справочника
- A. Практики (без ПО)
- B. Классы решений (BPM/workflow, RPA, OCR/IDP, ECM/СЭД, case management, BI, process mining, knowledge management, chatbot/copilot, AML/KYC/regtech, GRC/controls, integration/iPaaS)
- C. Конкретные продукты (подтверждённые capability)
- D. Компании (разработчики/интеграторы/сервис)
- E. Кейсы/доказательства

## Поля функции/процесса, нужные для будущего мэтчинга (собирать заранее)
частота; объём; SLA/критичность; доля ручного труда; доля операций по правилам;
доля экспертного суждения; число исключений; входные документы/данные; выходы;
текущие системы; число участников/согласований; требования к аудиту; чувствительность данных;
регуляторные ограничения; explainability; AI допускается или только deterministic;
on-prem/cloud; требуемые интеграции.

## Мэтчинг (прозрачный score, не «магия ИИ»)
5 блоков: релевантность функции · соответствие ограничениям · ожидаемый эффект ·
сложность внедрения · качество доказательной базы.
Итог: **подходит / условно подходит / не подходит** + объяснение.

## Фильтры (заложить сразу)
российское/иностранное; наличие в РФ; on-prem/cloud/hybrid; поддержка русского;
интеграция с текущим стеком; локальный партнёр; реестр отечественного ПО;
AI yes/no; human-in-the-loop; объяснимость; защита данных; отрасль/размер.

## Место в траектории
1. Функции и оргструктура ✅
2. Процессы и параметры функции
3. **База практик и решений** ← этот ADR
4. Мэтчинг
5. Shortlist и дорожная карта пилотов

Запуск наполнения — **после стабилизации функций**, параллельно процессной детализации.

## План реализации (порядок)
1. ADR / каркас (этот документ) — **сделано сейчас**
2. Шаблоны `data/templates/` для базы решений (реестр источников + extract sheet), controlled-load как у функций
3. Модель данных (таблицы сущностей)
4. UI: модуль «Практики и решения» + точка входа «Подобрать практики и решения» из карточки функции
5. Мэтчинг и shortlist

## Не делаем сейчас
Полноценную реализацию вширь. Только фиксация архитектуры, чтобы вернуться без потери качества.
