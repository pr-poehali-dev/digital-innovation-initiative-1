# Stage 15 — Hypotheses without validation queue

## Контекст решения
Изначально рассматривался вариант `solutions without_system`, но модель данных его не подтвердила: «решение» и «система» в проекте — одна и та же сущность (таблица `workspace_solutions` / вкладка «Решения и системы»), отдельного поля/сущности «система» нет. Раздувать модель данных ради одного preset — плохой trade-off.

Вместо этого выбран **data-driven** путь: у гипотез (`workspace_hypotheses`) уже есть честное поле `status` со значениями `open` / `testing` / `confirmed` / `rejected`, по умолчанию `'open'`. Это реальное незаполненное состояние, а не искусственно введённое.

## Что это
Preset `without_validation` — гипотезы, которые ещё не начали проверяться (`status === "open"`).

Продолжает воронку:
```
pain → hypothesis → solution
Stage 13   Stage 14
```
Stage 15 идёт вглубь другого измерения — качества самих гипотез, а не структурной связанности:
```
hypothesis (open) → взята в проверку (testing/confirmed/rejected)
```

Не пересекается с `without_initiative` (Stage 7/11): гипотеза может одновременно быть `open` и без инициативы — это не конфликт, а два разных диагноза, ведущих в разные, не дублирующие очереди.

## Что изменилось
- Виджет overview **«Гипотезы без проверки»** → `tab=hypotheses&preset=without_validation`
- На вкладке «Гипотезы»: preset-chip, счётчик «Осталось: N», completion-state «Все гипотезы взяты в проверку»
- Бейдж на карточке: **«Не проверена»**
- Усиленный CTA вместо обычной кнопки статуса: **«Начать проверку»** (переводит гипотезу в `testing`)
- После действия — success-feedback «Статус обновлён», фокус на следующей карточке в очереди, либо completion-state

## Out of scope
- Не вводится отдельная сущность/поле «система»
- Не меняется схема БД
- Не создаётся отдельный validation-lifecycle (evidence, confidence, эксперименты) — используется существующее поле `status`
